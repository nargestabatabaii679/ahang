# songai 🎵

هدیه‌ای که با صدای خودش می‌خواند، یک آهنگ شخصی، با صدای واقعیِ کسی که دوستش داری، که از روی یک عکس و یک نمونهٔ صدا ساخته می‌شود.

## ۰۰۰۰۰۰۰۰۰. پاس یازدهم — رفع باگ «جنریت خطا می‌دهد و چیزی ساخته نمی‌شود»

### علت اصلی
از ۵ مرحلهٔ پایپ‌لاین، فقط ۲ تا (متن ترانه، ویدیو) یک fallback امن داشتند — اگر سرویس بیرونی پیکربندی نشده بود، به‌جای کرش‌کردنِ کل سفارش، با کیفیت دمو ادامه می‌دادند. مراحل **موسیقی** و **صدا** این fallback را نداشتند: اگر `RIFFUSION_API_BASE` در دسترس نبود یا `ELEVENLABS_API_KEY` تنظیم نشده بود، خطا مستقیم از آن مرحله بالا می‌آمد و کل سفارش با وضعیت «خطا» متوقف می‌شد — دقیقاً همان چیزی که گزارش شد. این حالت وقتی پیش می‌آید که پروژه روی یک سرور تازه (بدون پیکربندی هیچ‌کدام از ۴ سرویس بیرونی) اجرا شود و `PIPELINE_MOCK=true` هم تنظیم نشده باشد.

### رفع
مرحلهٔ موسیقی و مرحلهٔ صدا حالا همان الگوی مراحل دیگر را دارند: تلاش با سرویس واقعی، و اگر شکست خورد (هر دلیلی — کلید نبود، سرویس در دسترس نبود، شبکه قطع بود)، به‌جای متوقف‌کردن کل سفارش، با یک هشدار در لاگ سرور به نسخهٔ دمو سوییچ می‌کنند (موسیقیِ جایگزین + همان نمونه‌صدای آپلودی کاربر). نتیجه: **سفارش همیشه به نتیجه می‌رسد**، چه همهٔ سرویس‌ها پیکربندی شده باشند چه هیچ‌کدام.

### یک باگ واقعیِ دوم که در همین بررسی پیدا شد
`cloneVoice` در ElevenLabs همیشه فایل صدا را با نوع `audio/mpeg` می‌فرستاد، صرف‌نظر از فرمت واقعی فایل — یعنی ضبط‌های مرورگر (`.webm`) یا فایل‌های آپلودی غیر-mp3 (`.wav`, `.m4a`, `.ogg`) با نوع محتوای اشتباه به API ارسال می‌شدند. یک تابع `mimeFromExt` اضافه شد تا نوع واقعی فایل از روی پسوندش استخراج و درست ارسال شود.

### تست
با **پاک‌کردن کامل `.env.local`** (یعنی دقیقاً سناریوی «هیچ سرویسی پیکربندی نشده») کل فلو را با Playwright اجرا کردم — لاگ سرور را بررسی کردم و این دو خط واقعاً ثبت شدند:
```
[pipeline] music provider failed (fetch failed); falling back to placeholder instrumental
[pipeline] voice clone failed (ELEVENLABS_API_KEY تنظیم نشده است); falling back to the raw uploaded sample
```
و سفارش با موفقیت به نتیجهٔ نهایی رسید — یک فایل ویدیوی واقعی و معتبر ساخته شد (تأیید شد با `ffprobe`)، نه خطا.

### تغییر متن صفحهٔ اول
طبق درخواست، تیتر و زیرتیتر هیرو به این تغییر کرد:
- تیتر: «یک هدیه که فراموش نمی‌شود»
- زیرتیتر: «با عکس و صدای خودت، یک آهنگ و ویدیوی شخصی بساز؛ هدیه‌ای که فقط برای یک نفر ساخته شده است.»

## ۰۰۰۰۰۰۰۰۰. پاس یازدهم — تطبیق با مسیر کاربریِ دقیق + صفحهٔ اشتراک‌گذاری

بیشتر این مسیر کاربری (۵ مرحله: گیرنده → حال‌وهوا → خودت را وارد کن → ساخت → نتیجه) از پاس‌های قبلی از قبل پیاده‌سازی شده بود و با تست زنده (اسکرین‌شات هر مرحله) تأیید شد که دقیقاً مطابق spec است — حتی متن بستن («امیدواریم این آهنگ، لبخند روی لبش بیاورد») و کپشن روی ویدیو («برای «سارا» ❤️») از قبل عین متن درخواستی بودند.

یک شکاف واقعی در بررسی پیدا شد: دکمه‌های اشتراک‌گذاری (واتساپ/تلگرام/استوری/کپی لینک) مستقیم به فایل خام ویدیو/صدا (`/generated/xxx-video.mp4`) لینک می‌دادند، نه یک صفحهٔ واقعی. این یعنی در واتساپ/تلگرام هیچ پیش‌نمایش زیبایی نشان داده نمی‌شد (اکثر پلتفرم‌ها لینک mp4 خام را به‌شکل یک آیکون فایل عمومی نشان می‌دهند، نه پخش‌کنندهٔ غنی).

رفع شد با یک صفحهٔ تازه: **`/gift/[id]`** — صفحهٔ عمومیِ اشتراک‌گذاری با:
- متادیتای Open Graph واقعی (`og:title`, `og:image` از روی کاور اسپکتروگرام، `og:video`) برای پیش‌نمایش درست در واتساپ/تلگرام/X
- یک دکمهٔ «بساز برای یکی» در پایین — یک حلقهٔ رشد طبیعی برای کسی که هدیه را گرفته
- ۴۰۴ سفارشیِ خودِ اپ اگر سفارش پیدا نشود یا هنوز آماده نباشد

دکمهٔ «کپی لینک» در `ResultView` حالا به‌جای فایل خام، به همین صفحه اشاره می‌کند.

تست شد: یک فلوی کامل با Playwright تا کلیک روی «کپی لینک»، خواندن واقعیِ کلیپ‌بورد، رفتن به همان لینک، و بررسیِ متاتگ‌های `og:image`/`og:video` (هر دو URL مطلق و درست) — به‌علاوهٔ تست مسیر ۴۰۴ با یک شناسهٔ ساختگی.

## ۰۰۰۰۰۰۰۰. پاس دهم — تحلیل ۴ ماژول موسیقی و انتخاب بهترین گزینه

### تحلیل ساختار هر ماژول

| ماژول | چیست | چرا استفاده شد/نشد |
|---|---|---|
| **Songify** | یک اپ کامل اندروید (Kotlin/Gradle) برای پخش موسیقی | ❌ ابزار اشتباه؛ این یک پلیر است نه مولد، و حتی اگر مولد بود، استک فنی‌اش (اندروید native) با Next.js این پروژه هیچ ربطی ندارد |
| **claude-music** | موتور تولید موسیقی محلی روی ACE-Step 1.5، نیازمند GPU انویدیا (حداقل ۴ گیگ VRAM) و نصب پایتون روی همان ماشین | ❌ از نظر معماری با یک اپ وب جور درنمی‌آید: یعنی هر سرور Next.js باید یک GPU اختصاصی داشته باشد، که نه مقیاس‌پذیر است نه ساده. (ابزاری عالی برای Claude Code روی دسکتاپ شخصی، نه برای بک‌اند یک سرویس وب) |
| **Generative-Media-Skills** | اسکریپت‌های bash که به یک گیت‌وی شخص‌ثالث (muapi.ai) وصل می‌شوند؛ بخش موسیقی‌اش هم فقط یک wrapper روی همان Suno است | ❌ یک واسطهٔ تجاریِ اضافه برای همان قابلیتی که از قبل مستقیم از طریق `suno.ts` داریم؛ اضافه‌کردنش فقط یک نقطهٔ شکست و هزینهٔ بیشتر اضافه می‌کرد بدون قابلیت واقعاً تازه |
| **suno-song-creator-plugin** | دانشِ خالصِ prompt-engineering برای Suno: مرجع متاتگ‌ها، نقشهٔ «ابرهای ژانر» (genre clouds) و استراتژی فرار از ترکیب ناخواسته، بدون هیچ وابستگی زیرساختی تازه | ✅ **انتخاب شد** — دقیقاً منطبق بر چیزی که از قبل داریم (`suno.ts` به‌عنوان جایگزین Riffusion)، صفر وابستگی جدید، صفر هزینهٔ زیرساخت |

### چرا suno-song-creator-plugin برندهٔ بحث پرفورمنس/سادگی/مقیاس‌پذیری بود
- **پرفورمنس**: هیچ کدام از تکنیک‌های این اسکیل لیتنسی اضافه نمی‌کنند؛ فقط متن پرامپتی که از قبل به همان API ارسال می‌شد را بهتر می‌کنند.
- **سادگی**: صفر سرویس جدید، صفر کلید API جدید، صفر باینری جدید لازم است.
- **مقیاس‌پذیری**: چون `suno.ts` از قبل پشت یک سرویس HTTP خودمیزبان است، این تغییر فقط کیفیت خروجی همان مسیر را بالا می‌برد، بدون تغییر در معماری.

### یک باگِ واقعیِ بیلد-شکن که در همین بررسی پیدا شد
در حین مرور `lyrics.ts`، معلوم شد یک تلاش قبلی برای غنی‌سازی `buildMusicPrompt` (افزودن `tags`/`description` جدا به‌جای یک رشتهٔ ساده) نیمه‌کاره مانده بود: امضای تابع را عوض کرده بود ولی `suno.ts` و `pipeline.ts` هنوز انتظار یک رشتهٔ ساده داشتند. **`npm run build` واقعاً شکست می‌خورد.** این پاس آن را کامل و درست کرد:

- `generateMusic()` در `suno.ts` حالا همان آبجکت غنیِ `{tags, description}` را می‌گیرد (نه یک رشته)
- `tags` واقعی و مخصوص هر ژانر به API ارسال می‌شود (قبلاً یک رشتهٔ ثابت و عمومی `"instrumental, background music, no vocals"` برای همهٔ ژانرها فرستاده می‌شد و کاملاً نادیده گرفته می‌شد)
- یک فیلد `negative_tags` تازه اضافه شد — طبق مفهوم «چاهِ جاذبهٔ پاپ» اسکیل (تقریباً هر ژانر در Suno به‌سمت پاپ/مدرن می‌رود مگر صریحاً منع شود) و چون صدای آواز ما از کلون صدا می‌آید نه از Suno، هر اثر صدای خواننده در خروجیِ Suno یک نقص است، نه سلیقه — پس `Vocals, Singing, Rap Vocals, ...` همیشه exclude می‌شود

⚠️ **صادقانه**: نام فیلد `negative_tags` بر اساسِ مستندات شناخته‌شدهٔ gcui-art/suno-api است، ولی چون امکان تست مستقیم در برابر یک نمونهٔ زندهٔ این سرویس در این محیط نبود، قبل از استفادهٔ واقعی یک‌بار در برابر دیپلویمنت خودتان تأیید کنید.

`npm run build` بعد از این رفع باگ کاملاً تمیز رد می‌شود؛ کل فلو هم با Playwright دوباره از صفر تا دانلود کیتسیک تست شد.

## ۰۰۰۰۰۰۰. پاس نهم — ویرایش کپی + کیتسیک PDF فارسی

سه فایل آپلود شد؛ یکی‌شان (`inside-lago-voice-skill`) عمداً استفاده نشد — قالبی برای کدگذاری «صدای نوشتاری شخصیِ» یک فرد است (با خواندن ده‌ها نوشتهٔ دستیِ خودِ کاربر پر می‌شود) و من داده‌ای از صدای نوشتاری شما ندارم؛ ادعای پرکردنش بدون آن داده غیرصادقانه بود. دو تای دیگر مستقیم به‌کار رفتند:

### ۱) ویرایش کپی طبق اسکیل `copywriting` (از `agent-skills`)
حالت Edit این اسکیل می‌گوید خط‌تیرهٔ بلند (—) «قابل‌اعتمادترین ردپای AI» در متن است (حداکثر ۱ در هر ۱۰۰۰ کلمه). ممیزی کل کپی‌های کاربر-محورِ اپ را گرفت: **۹ مورد** در رشته‌های UI (صفحهٔ اول، فرم رضایت، ویس‌اینپوت، alt تصویر) و **۵۱ مورد** در همین README — همه‌شان از من، در پاس‌های قبلی. همه با علامت‌های فارسیِ مناسب جایگزین شدند (ویرگول، نقطه‌ویرگول، یا نقطه برای تیترها) به‌جای حذف کورکورانه.

### ۲) کیتسیک PDF فارسی با xepersian (از `Persian-LaTeX-Guide-and-Examples`)
یک قابلیت تازه: دکمهٔ «دانلود کیتسیک متن ترانه» در صفحهٔ نتیجه، که یک PDF با تایپوگرافی واقعیِ فارسی (نه HTML-to-PDF) از متن ترانه می‌سازد — به‌صورت دستی تست شد، نه فقط نوشته شد:

- **فونت**: بسته‌بندی فونت‌های XB Zar در آن ریپو فاقد فایل LICENSE بود، پس به‌جایش از فونت **Amiri** استفاده شد (همان فونتی که با خود بستهٔ `texlive-lang-arabic` نصب می‌شود، مجوز SIL OFL، کاملاً روشن).
- **یک باگ واقعی در حین تست پیدا شد**: نسخهٔ اول قالب از `tikz` برای یک قاب تزئینی دور صفحه استفاده می‌کرد. کامپایل بدون خطا تمام می‌شد، ولی PDF خروجی کاملاً سفید بود (`tikz overlay` با پکیج `bidi` که `xepersian` زیر پوستش استفاده می‌کند ناسازگار است). با حذف tikz و تکیه بر تایپوگرافی/فاصله‌گذاری به‌جای قاب، مشکل رفع شد.
- **یک باگ دوم هم پیدا شد**: ترتیب `\usepackage` مهم است؛ `geometry` باید قبل از `xepersian` لود شود وگرنه `bidi` خطا می‌دهد.
- **امنیت**: نام گیرنده، مناسبت، و متن ترانه همگی از کاربر می‌آیند و مستقیم در فایل `.tex`ی نوشته می‌شوند که بعد `xelatex` آن را کامپایل می‌کند. یک تابع `escapeLatex` همهٔ کاراکترهای خاص LaTeX را خنثی می‌کند (و خودِ بک‌اسلش را اول از همه، با یک placeholder، تا مراحل بعدی اسکیپ‌کردن، خروجیِ خودش را خراب نکنند — یک باگ واقعی که تست با ورودیِ `\input{x}` آن را لو داد). `xelatex` هم با فلگ `-no-shell-escape` اجرا می‌شود تا حتی در صورت دور زدن اسکیپ، اجرای دستور سیستمی ممکن نباشد. این را با ورودیِ عمداً خصمانه (`\input{x}`, `$`, `&`, `#`, `~`, `^`, `{`, `}`) تست کردم؛ همه به‌صورت متن بی‌خطر چاپ شدند، نه دستور.
- **محدودیت صادقانه**: این قابلیت نیاز به نصب‌بودن `xelatex` (از `texlive-xetex` + `texlive-lang-arabic`) روی سروری دارد که Next.js را اجرا می‌کند. روی پلتفرم‌های serverless مثل Vercel این بسته‌ها نصب‌پذیر نیستند؛ روی یک VPS/کانتینر معمولی (همان‌جایی که ffmpeg هم لازم است) مشکلی نیست. اگر `xelatex` نصب نباشد، دکمه خطای روشنی نشان می‌دهد، نه کرش.

تست کامل: یک فلوی end-to-end واقعی با Playwright — از پر کردن فرم تا دانلود واقعیِ فایل PDF و رندر آن — اجرا و فایل خروجی به‌صورت تصویری بررسی شد.

## ۰۰۰۰۰۰. پاس هشتم · ارتقای اواتار طبق اسکیل «ai-avatar-video»

این اسکیل عمدتاً برای ویدیوهای تبلیغاتی/UGC با اواتار است (اسکریپت hook-to-CTA، انتخاب پرسونا برای فروش، لحن UGC)، این بخش‌ها عمداً استفاده **نشدند**، چون songai یک هدیهٔ شخصی است (صدا/چهرهٔ واقعیِ یک عزیز، نه یک پرسونای تبلیغاتی) و زبان فروش/CTA در این زمینه نامناسب است. دو بخش واقعاً قابل‌استفاده بودند:

### ۱) Creatify Aurora به‌عنوان موتور دومِ اواتار
بخش ۲٫۴ اسکیل دقیقاً همان کاری را توضیح می‌دهد که songai نیاز دارد: «عکس + صدا → ویدیوی اواتار با بهترین کیفیت لب‌سینک». این معماری دقیقاً معادل OmniHuman ماست، پس آن را به‌عنوان یک گزینهٔ موازی اضافه کردم:
- `src/lib/providers/creatify.ts` (تازه)، همان الگوی create+poll که در `runcomfy.ts` و `suno.ts` دیده‌اید.
- سوئیچ `VIDEO_PROVIDER` حالا سه مقدار می‌پذیرد: `omnihuman` (پیش‌فرض) · `kling` · `aurora`.
- چرا لازم بود: اگر RunComfy در دسترس نباشد یا بخواهید کیفیت لب‌سینک Aurora را امتحان کنید، بدون تغییر کد فقط با عوض‌کردن یک متغیر محیطی سوییچ می‌کنید.

### ۲) راهنمای تلفظ برای صدای کلون‌شده
بخش «Pronunciation Guidance» اسکیل می‌گوید برای TTS باید اعداد را به‌صورت کلمه نوشت و از مخفف لاتین پرهیز کرد چون صدای مصنوعی این‌ها را اشتباه می‌خواند. این مستقیماً به مشکل واقعی ما می‌خورد: ترانه‌ای که هوش مصنوعی می‌نویسد، قرار است توسط صدای کلون‌شدهٔ کاربر **خوانده** شود، اگر شعر شامل رقم یا حروف لاتین باشد، خروجی نامفهوم می‌شود. این قانون به `SYSTEM_PROMPT` در `src/lib/providers/lyrics.ts` اضافه شد: مدل دیگر رقم یا مخفف لاتین در شعر نمی‌نویسد.

### تست
- `npm run build` بدون خطا.
- یک تست end-to-end کامل با Playwright (صفحهٔ اول → فرم → آپلود → رضایت → تولید → نتیجه → ۴۰۴ → حریم خصوصی → شرایط استفاده) بدون خطای کنسول واقعی اجرا شد.
- فایل‌های خروجی (`ffprobe`) و پاسخ‌های API (`curl`) دوباره صحت‌سنجی شدند.

## ۰۰۰۰۰. پاس هفتم · ارتقای میکس صدا طبق اسکیل «avatar-mix»

اسکیل `avatar-mix` یک پایپ‌لاین کاملاً متفاوت است (اواتار HeyGen چندصحنه‌ای با صحنه‌های fullscreen/corner/bg_only، زیرنویس، انتشار در شبکه‌های اجتماعی) و مستقیماً روی معماری ویدیوی songai قابل اعمال نبود، اما اسکریپت `composite.py` همان اسکیل، دو تکنیک مهندسی صدای واقعی داشت که دقیقاً به نقطه‌ضعف موجود در `ffmpeg.ts` می‌خورد:

| قبل | بعد |
|---|---|
| موسیقی با یک `volume=0.45` ثابت در کل طول آهنگ کم می‌شد، چه صدا در حال خواندن باشد چه سکوت | **sidechain ducking واقعی** (`sidechaincompress`): موسیقی فقط وقتی صدای خواننده واقعاً در حال پخش است کم می‌شود؛ در سکوت‌های بین خط‌ها، موسیقی نفس می‌کشد |
| فقط `dynaudnorm` برای نرمال‌سازی، بدون محافظت در برابر کلیپ شدن | یک `alimiter` در انتهای زنجیره میکس اضافه شد (دقیقاً مقدار `limit=0.95` از خودِ اسکیل) |
| فایل‌های mp4/mp3 خروجی، متادیتای پیش‌فرض ffmpeg (نسخهٔ انکودر، creation_time) را همراه داشتند | `-map_metadata -1` به همهٔ خروجی‌ها اضافه شد؛ یک تابع جدید `stripVideoMetadata` هم برای پاک‌سازی ویدیوی دانلودشده از RunComfy/OmniHuman (بدون encode دوباره، با `-c copy`) اضافه شد، دقیقاً همان دستور `scripts/strip_meta.sh` از اسکیل |
| بدون `+faststart` | اضافه شد، ویدیو قبل از دانلود کامل شروع به پخش می‌کند |

### تست
هر دو filter-graph (sidechain ducking و metadata-stripping) ابتدا مستقیم با ffmpeg روی فایل‌های نمونه تست شدند، سپس کل فلوی برنامه با Playwright از صفحهٔ اول تا نتیجهٔ نهایی اجرا شد:
- صفحهٔ اول، صفحهٔ ساخت، صفحهٔ ۴۰۴، حریم خصوصی و شرایط استفاده، همه بدون خطای کنسول بررسی شدند (تنها خطاهای واقعی، ۴۰۳ از CDN فونت بود که محدودیت شبکهٔ همین محیط تست است، نه باگ اپ).
- فایل‌های خروجی واقعی (`*-video.mp4`, `*-final.mp3`) با `ffprobe` بررسی شدند: کدک، طول، و عدم وجود متادیتای اضافه تأیید شد.
- اعتبارسنجی سرور برای رضایت (`consent`) و درخواست‌های ناقص/بدفرم با `curl` مستقیم تست شد، هر دو خطای JSON تمیز برمی‌گردانند، نه کرش.

## ۰۰۰۰. پاس ششم · ممیزی موشن طبق اسکیل‌های انیمیشن

از چهار فایل آپلودی، دو تا مستقیماً استفاده شدند چون چک‌لیست‌های دقیق و قابل‌اجرا داشتند: **design-motion-principles** (ممیزی الگوهای «AI-slop motion») و **ui-animation** از `agent-skills` (دستور‌العمل‌های دقیق timing/easing). **animation-principles** (اصول دیزنی، کلی‌تر و کم‌دقت‌تر از این دو) و **motion-dev-animations-skill** (عمدتاً قالب برای ساخت کامپوننت از صفر) چیز تازه‌ای اضافه نکردند و کنار گذاشته شدند.

### باگ واقعی دسترس‌پذیری که پیدا شد
قانون سراسری `prefers-reduced-motion` در `globals.css` فقط روی **CSS** animation/transition اثر می‌گذارد، انیمیشن‌های framer-motion (که با JS اجرا می‌شوند: گذار بین مراحل ویزارد، پاپ‌شدن چیپ‌های انتخاب، ریویل صفحهٔ نتیجه) اصلاً به این قانون توجه نمی‌کردند. یعنی کاربری که در سیستم‌عاملش «کاهش حرکت» را فعال کرده، همچنان همهٔ این انیمیشن‌ها را کامل می‌دید. رفع شد با پوشاندن کل اپ در `<MotionConfig reducedMotion="user">` (`src/components/MotionProvider.tsx`)، حالا هر انیمیشن framer-motion در سراسر اپ به‌صورت خودکار از این تنظیم پیروی می‌کند.

### الگوهای «AI-slop motion» که پیدا و رفع شد

| الگو | کجا | رفع |
|---|---|---|
| آیکون «اسپارکل» در صفحهٔ نتیجه بی‌وقفه نفس می‌کشید (`animate-breathe`) | `ResultView.tsx` | حذف شد، طبق اسکیل: «pulsing/breathing روی هر چیزی پیش‌فرض باید static باشد مگر کاربر صریحاً پالس بخواهد». ریویل یک‌بارهٔ خود کارت برای این لحظه کافی است |

| لینک Skip-to-content با `top`/`-top-20` انیمیت می‌شد، یک ویژگی لایوت، نه transform | `layout.tsx` | به `translateY` تغییر کرد (GPU-accelerated، بدون reflow) |

| دکمهٔ پلاس‌تایمینگ ۳۰۰ms برای hover/press، کندتر از بازهٔ توصیه‌شده | `button.tsx` | به ۲۰۰ms (در بازهٔ «hover ساده: ۲۰۰ms» از جدول اسکیل) تغییر کرد |
| پیام خطای فرم فقط رنگ قرمز می‌گرفت، بدون بازخورد حرکتی | `CreateFlow.tsx` | الگوی نام‌دارِ «error state shake» از `ui-animation` پیاده شد (یک لرزش ۲۲۰ms با `cubic-bezier(0.36,0.07,0.19,0.97)`)، با `key={error}` تا هر پیام جدید دوباره اجرا شود |
| دات قرمزِ «در حال ضبط» با `animate-pulse` | `VoiceInput.tsx` | نگه داشته شد، چون استثنای صریح اسکیل است: «یک نمونهٔ تنها، با دلیل طراحی مکتوب»، اینجا وضعیت واقعی میکروفون را نشان می‌دهد، نه پالس تزئینی. دلیلش در کد کامنت شد |
| کیفریم‌های مرده (`breathe`, `shimmer`, `spin-slow`) که هیچ‌جا استفاده نمی‌شدند | `tailwind.config.ts` | حذف شدند |

## ۰۰۰. پاس پنجم · ارتقای بخش ویدیو طبق اسکیل «lipsync»

از چهار فایل آپلودی، فقط `SKILL.md` (اسکیل `lipsync` برای RunComfy CLI) محتوای عملی و کامل داشت، بقیه (`video-prompting-skill`, `bencium-marketplace`) راهنمای مدل‌های متن‌به‌ویدیوی عمومی یا فهرست مارکت‌پلیس بودند و به مرحلهٔ ویدیوی songai (که avatar از روی عکس می‌سازد، نه صحنه از صفر) ربط مستقیم نداشتند، پس استفاده نشدند. `LibreUIUX` هم یک مجموعهٔ عمومی UX بود که چیز اختصاصی‌ای به این مرحله اضافه نمی‌کرد.

اسکیل `lipsync` یک باگ معماری واقعی را آشکار کرد: کد قبلی (`kling.ts`) به Kling لیپ‌سینک با `image_url` درخواست می‌داد، اما طبق خودِ اسکیل، مسیر Kling/Sync Labs برای زمانی‌ست که یک **ویدیوی منبع** داری و فقط لب را هماهنگ می‌کنی، نه یک عکس ثابت. ورودی واقعی songai («یک عکس پرتره + صدا») دقیقاً مسیر دیگری از همین اسکیل است: **ByteDance OmniHuman** (آواتار گویا از روی پرتره). این یعنی پیاده‌سازی قبلی، در اولین استفادهٔ واقعی، با خطای schema شکست می‌خورد.

تغییرات:
- `src/lib/providers/kling.ts` (دارای باگ) حذف شد؛ جایگزین آن `src/lib/providers/runcomfy.ts` است که دقیقاً همان فراخوانی مستندشدهٔ اسکیل را اجرا می‌کند: `runcomfy run bytedance/omnihuman/api --input '...' --output-dir ...`، با تفسیر کدهای خروج (۶۴/۶۵/۶۹/۷۵/۷۷) طبق جدول خودِ اسکیل.
- سوئیچ `VIDEO_PROVIDER=omnihuman|kling` (الگوی همان سوئیچ `MUSIC_PROVIDER` از پاس قبلی) اضافه شد.
- **رضایت صریح**: اسکیل تأکید می‌کند «درایوکردن دهانِ یک فرد واقعی از یک فایل صدای جدا، دو-کاربردی است» و عملیات باید از سمت اپراتور تأیید شود. یک چک‌باکس رضایت واقعی (نه فقط متن در فوتر) قبل از دکمهٔ «بساز» اضافه شد که تا تیک نخورد، دکمه غیرفعال است، هم سمت کلاینت و هم سمت سرور (`api/jobs/route.ts` بدون `consent=true` درخواست را با خطا رد می‌کند).
- caption مرحلهٔ «ساخت ویدیو» در نوار پیشرفت به‌روزرسانی شد تا OmniHuman را به‌جای Kling نشان دهد.

تست شد: یک فلوی کامل با Playwright (پر کردن فرم → آپلود → تلاش برای ارسال بدون تیک رضایت [دکمه غیرفعال بود] → تیک‌زدن [دکمه فعال شد] → ساخت تا نتیجهٔ نهایی) با موفقیت اجرا شد.



این نسخه شامل یک بازطراحی کامل بصری (برای مخاطب ۱۶ تا ۳۵ سال)، رفع باگ‌های واقعی، و آماده‌سازی برای انتشار به‌صورت اپلیکیشن نصب‌شدنی در App Store و Google Play است. جزئیات هرکدام پایین آمده.

## ۰۰. پاس چهارم · ارتقای بخش موسیقی با Riffusion

موتور ساخت موسیقی از یک API جعبه‌سیاه (Suno) به **Riffusion** تغییر کرد، کتابخانه‌ای که موسیقی را با اجرای Stable Diffusion روی *تصویر اسپکتروگرام* و درون‌یابی (interpolation) بین دو پرامپت متنی می‌سازد. این یعنی:

- **هر سبک موزیک یک سفر دو نقطه‌ای واقعی دارد**, نه یک رشتهٔ تگ ساده. مثلاً برای «سنتی»: شروع از «ستار و دف، فولک صمیمی» و پایان در «سنتور و تنبک، گرمای کلاسیک ایرانی». هر ۶ سبک در `src/lib/providers/riffusion.ts` این‌طور تعریف شده‌اند.
- **اسپکتروگرامِ واقعی مدل به‌عنوان کاور آهنگ نگه داشته می‌شود.** Riffusion علاوه بر صدا، همان تصویری که روی آن diffusion زده را هم برمی‌گرداند؛ این تصویر به‌جای یک آیکون ژنریک، به‌عنوان `poster` ویدیوی نهایی و کاور تیکت ترانه استفاده می‌شود، یعنی تزیین نیست، خروجی واقعی مدل است.
- **حالت mock هم از تصاویر واقعی Riffusion استفاده می‌کند**: ۵ تا از seed imageهای خودِ کتابخانه (`public/riffusion-seeds/`, مجوز MIT) به‌عنوان کاور دمو در حالت `PIPELINE_MOCK=true` نمایش داده می‌شوند، که برای هر سبک یکی انتخاب می‌شود.
- **سوئیچ `MUSIC_PROVIDER`** بین `riffusion` (پیش‌فرض) و `suno` (همان آداپتور قبلی، هنوز موجود است) جابه‌جا می‌کند، بدون نیاز به تغییر کد.

### راه‌اندازی سرور Riffusion (یک سرویس پایتون جدا)
```bash
git clone https://github.com/riffusion/riffusion riffusion-server
cd riffusion-server && python -m pip install -r requirements.txt
python -m riffusion.server --host 0.0.0.0 --port 3013
```
سپس در `.env.local`:
```
MUSIC_PROVIDER=riffusion
RIFFUSION_API_BASE=http://localhost:3013
```
⚠️ برای تولید نزدیک به بلادرنگ به GPU نیاز دارد (مثل 3090 یا A10G)؛ روی CPU هم کار می‌کند ولی کند است.

### باگ واقعی که در همین پاس پیدا و رفع شد
هنگام تست end-to-end (با Playwright)، یک خطای واقعی در کنسول مرورگر افتاد:
`The radius provided (-1.96429) is negative.`
علتش این بود که در `Waveform.tsx`، وقتی canvas هنوز عرضش صفر/خیلی کم اندازه‌گیری شده بود (مثلاً موج کوچک داخل تیکت، قبل از layout کامل)، عرض هر میله منفی می‌شد و شعاع گردکردن گوشه‌ها هم منفی می‌شد، که باعث می‌شد `roundRect` در canvas خطا بدهد. رفع شد با محدود کردن عرض میله و شعاع به حداقل صفر.

## ۱. فلسفهٔ طراحی · «میکس‌تیپ» (نسخهٔ دوم)

این نسخه یک پاس طراحی جدید روی برند است؛ هویت قبلی («شب کاشان»، نیلی + طلا + فیروزه) به یک مفهوم متفاوت و ریسک‌پذیرتر تغییر کرد چون آن نسخه هنوز به الگوی رایج «AI-generated» (پس‌زمینهٔ تیره + یک رنگ تاکیدی روشن) نزدیک بود. مفهوم جدید مستقیماً از خودِ محصول می‌آید:

**ایده**: این اپ یک «نوار کاست» شخصی می‌سازد، پس چرا شبیه یک اپ SaaS باشد؟ تمام زبان بصری حالا حول مفهوم نوار کاست/میکس‌تیپ دست‌ساز می‌چرخد.

**پالت** (۵ رنگ نام‌گذاری‌شده):
- `#150F22` ink، اتاقِ شب، رنگ زمینه (بنفش‌تیرهٔ گرم، نه مشکی مطلق)
- `#FFB23F` tape، نور کهربایی دکِ ضبط (رنگ اصلی)
- `#FF6F61` thread، نخ مرجانی که تیکت را می‌دوزد (رنگ دوم/تاکیدی)
- `#F6ECD9` paper، فقط برای سطح تیکت/برچسب، نه پس‌زمینهٔ کلی
- `#2B2118` متن روی تیکت

**تایپوگرافی**: «وزیرمتن» در وزن ۸۰۰ برای تیترها (به‌جای آوردن یک فونت نمایشی جدید و نچسب)، و «JetBrains Mono» فقط برای عددهای روی تیکت، دقیقاً مثل شمارندهٔ مکانیکی روی دکِ کاست. این یک جفتِ تایپوگرافی با معنا است، نه یک انتخاب دلخواه.

**عنصر امضادار**: «تیکت میکس‌تیپ» (`.ticket` در `globals.css`)، یک کارت کاغذی با لبهٔ پرفراژ (نقطه‌چین) و دو سوراخ گرد در دو سر، دقیقاً مثل برچسب کاست‌هایی که با دست می‌نوشتند. این عنصر یک‌بار در صفحهٔ اول (با اسم «سارا» به‌عنوان نمونه) و یک‌بار در صفحهٔ نتیجه (با متن واقعی ترانه) دیده می‌شود، یعنی قول صفحهٔ اول و چیزی که در پایان تحویل کاربر داده می‌شود، تصویر یکسانی دارند.

**ساختار**: بخش «در سه قدم» حالا «Side A» نام‌گذاری شده و شماره‌ها به‌جای دایره‌های معمولی ۰۱/۰۲/۰۳، به سبک شمارندهٔ نواری (مونو، کم‌رنگ، گوشهٔ کارت) نمایش داده می‌شوند، چون اینجا ترتیب واقعاً معنا دارد (هر قدم، قدم بعد را باز می‌کند).

فایل‌های کلیدی:
- `src/app/globals.css`، توکن‌های رنگ/فونت + کلاس‌های `.ticket`, `.ticket-perf`, `.ticket-notch`, `.thread-divider`
- `tailwind.config.ts`، پالت `tape` / `thread` / `paper`
- `src/app/page.tsx`، ترکیب‌بندی هیرو حول تیکت
- `src/components/result/ResultView.tsx`، تکرار تیکت در صفحهٔ نتیجه

## ۳. آماده‌سازی برای App Store و Google Play

⚠️ **نکتهٔ مهم و صادقانه**: این پروژه یک اپ وب Next.js است، نه یک پروژهٔ native iOS/Android. برای انتشار واقعی در استورها باید آن را در یک پوستهٔ نصب‌شدنی بسته‌بندی کنید. خودِ اپ از نظر فنی برای این کار آماده شده (PWA کامل با مانیفست، آیکون و safe-area)، ولی پکیج‌کردن نهایی برای استور یک مرحلهٔ جداگانه است که باید روی سیستم خودتان (با Xcode / Android Studio) انجام شود، چون این محیط دسترسی به ابزارهای ساخت native ندارد.

دو مسیر استاندارد و رایج:

### الف) Google Play · با Trusted Web Activity (TWA)
ساده‌ترین و رایج‌ترین مسیر برای اپ‌های Next.js:
```bash
npm install -g @bubblewrap/cli
bubblewrap init --manifest https://your-domain.com/manifest.webmanifest
bubblewrap build
```
این یک پروژهٔ اندرویدی می‌سازد که اپ شما را داخل یک WebView بدون نوار آدرس باز می‌کند. خروجی `.aab` مستقیماً قابل آپلود در Google Play Console است.

### ب) App Store (iOS) · با Capacitor
چون اپل از پذیرفتن یک WebView ساده سخت‌گیرتر است، بهتر است از Capacitor استفاده کنید تا پل native واقعی داشته باشید:
```bash
npm install @capacitor/core @capacitor/ios
npx cap init songai com.yourcompany.songai
npx cap add ios
npx cap open ios
```
سپس از Xcode برای ساخت Archive و ارسال به App Store Connect استفاده کنید. چون این اپ یک بک‌اند (API routes) دارد، ساده‌ترین حالت این است که آن را روی یک سرور واقعی دیپلوی کنید (مثلاً Vercel) و Capacitor را به همان دامنه وصل کنید.
